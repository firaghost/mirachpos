# Offline-to-Online Sync Analysis

This document outlines the investigation into the reported issues of **order inconsistency** and **order disappearance** within the POS application, specifically focusing on the offline-to-online synchronization logic during unstable internet connectivity.

## Overview of the Sync Architecture
The POS application (`mobile/src/screens/pos/PosScreen.tsx`) attempts to provide an "offline-first" experience by utilizing a local SQLite database (`mobile/src/lib/db.ts`). Operations that fail due to network errors, or operations performed when the device is explicitly offline, are queued into an `outbox` SQLite table via `enqueueOp`.

A background process (`processOutboxOnce` in `mobile/src/lib/offline/posSync.ts`) continuously reads this outbox and replays the operations (Create, Edit, Cancel, Pay) sequentially against the remote Supabase backend when the internet connection is restored.

## Key Findings & Root Causes

Based on a thorough review of the synchronization logic, the root cause of both "order disappearance" and "order inconsistency/duplication" is directly tied to critical flaws in the offline queue and synchronization mechanisms. 

### 1. Order Disappearance (Missing Local State)
When a user creates a new order while offline, the order appears to "disappear" from the POS immediately after creation.

**Why this happens:**
In `PosScreen.tsx`, when `isOnline` is false, the `placeOrder` function calls `enqueue` to push a `create` operation into the outbox, and then immediately calls `clear()` to clear the cart. 
However, the application **never** inserts the newly created order into the local SQLite `orders` table. Because `OrdersOverlay.tsx` reads from `getUnpaidOrdersLocal()` when offline, the newly enqueued order is effectively invisible to the waiter until the device reconnects to the internet, processes the outbox, and downloads the newly created order from Supabase via a realtime subscription or subsequent fetch.

### 2. Order Inconsistency & Duplication (Non-Idempotent Creates)
During unstable connections, the same order might appear multiple times on the server, or its items might be partially missing.

**Why this happens:**
The `performCreate` function in `mobile/src/lib/offline/posSync.ts` is not atomic and not idempotent. It executes sequentially:
1. `supabase.from('orders').insert(...)` -> Returns a new UUID
2. `supabase.from('order_items').insert(...)`

If the internet connection drops exactly after step 1 completes but before step 2 finishes, an exception is thrown (`itemsError`). The outbox processor catches this exception and increments the `try_count` for the `create` operation, leaving it in the queue.
When the internet reconnects, the system retries the **entire** `create` operation. This results in a second identical order being inserted into Supabase. The first order remains in the database with missing items, leading to severe order inconsistencies and ghost orders.

### 3. Queue Blocking on Updates (Missing ID Mapping)
When an order is created offline, it receives a temporary local ID generated by `newId()`. If the system were to allow editing of this offline order (e.g., if finding #1 was fixed), the edit operation would be queued referencing the local ID.

**Why this happens:**
When the outbox syncs, `performCreate` generates a brand new remote UUID from Supabase. It completely discards the local ID and fails to update the local SQLite database. Although an `id_map` table exists in `db.ts`, the sync logic in `posSync.ts` does not use it. Consequently, any subsequent `edit` or `pay` operations in the outbox queue would attempt to update Supabase using the unrecognized local ID, resulting in Supabase throwing an error. This error would cause the outbox processor to block infinitely on that operation, halting all future synchronizations.

### 4. Flawed Offline Payment Handling
If a payment is collected offline, the offline queue attempts to update the `payment_status` using `performPay`. 

**Why this happens:**
If `performPay` encounters an error, it contains a fallback to retry the update without the `payment_method`. However, just like `performCreate`, if the initial Supabase update succeeds but the subsequent insertions into `order_change_logs`, `guest_visits`, or table status updates fail due to a connection drop, the operation will be retried, potentially duplicating audit logs and guest visit records.

## Conclusion

The hypothesis is **correct**. The reported order inconsistencies and disappearances are unequivocally caused by architectural flaws in the offline-to-online syncing process. 

To resolve these issues, the system requires:
1. **Local Optimistic Updates:** Offline orders must be written to the local SQLite `orders` and `order_items` tables immediately so they remain visible and editable.
2. **Idempotency Keys / ID Mapping:** The `posSync.ts` processor must map local IDs to remote UUIDs using the `id_map` table, and Supabase must use idempotency keys (or upserts using predefined local IDs if UUIDs are generated client-side) to prevent duplication during network timeouts.
3. **Atomic Sync Operations:** Multi-step Supabase inserts (order + items + audit logs) must either be consolidated into a single RPC call or handled with strict rollback/retry logic to prevent partial data states.
